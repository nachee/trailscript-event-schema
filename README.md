# trailscript-event-schema [![CI](https://github.com/nachee/trailscript-event-schema/actions/workflows/ci.yml/badge.svg)](https://github.com/nachee/trailscript-event-schema/actions/workflows/ci.yml)

The canonical schema for user-interaction **events** and **DOM checkpoints** — one Zod source of
truth, compiled to **JSON Schema** and **Pydantic v2** models so a TypeScript tracker and a Python
analysis service validate against the exact same contract.

This is the shared boundary type extracted from **TrailScript**, an automated E2E-test-generation
product. A browser tracker ([`trailscript-tracker`](https://github.com/nachee/trailscript-tracker))
emits events in this shape; the analysis pipeline
([`trailscript-testgen`](https://github.com/nachee/trailscript-testgen)) reads them back typed.
Defining that contract once, in one language, and generating the rest keeps the two ends from
drifting.

```
        src/*.ts  (Zod — single source of truth)
            │
   ┌────────┴─────────┐
   ▼                  ▼
lib/*.js         dist/*.schema.json
(TS types +      (JSON Schema)
 runtime            │
 validation)        ▼
             dist/python/event_schema/*.py
             (Pydantic v2 models)
```

## Why it's interesting

- **One source, three artifacts.** The Zod schemas in `src/` are authoritative. `npm run build`
  emits both runtime-validating TS (`lib/`) and JSON Schema (`dist/`); `make python-codegen` turns
  the JSON Schema into Pydantic v2 models. No hand-written duplicate types on either side.
- **Discriminated unions, fully covered.** Events are a discriminated union over `event_type`
  (click, fill, navigate, select, check, …), each with its own payload shape (`src/payloads.ts`).
  Tests exercise every variant at 100% coverage.
- **Codegen sharp edges handled.** `datamodel-codegen` emits an invalid `model_config` on Pydantic
  `RootModel` subclasses; `scripts/fix_rootmodel_config.py` strips it as a post-process so the
  generated package imports cleanly. The JSON Schema build deliberately names the root schema for
  readable generated Python class names (`Event1…Event32`, not `Model1…`).

## Usage

### TypeScript

```bash
npm install
npm run build
```

```typescript
import { EventSchema } from "trailscript-event-schema";

const result = EventSchema.safeParse(incoming);
if (result.success) {
  // result.data is fully typed
}
```

JSON Schema is exported for non-TS consumers:

```typescript
import eventJsonSchema from "trailscript-event-schema/json-schema";
```

### Python

The generated Pydantic package lives in `dist/python/` (committed, and regenerated by the
Makefile):

```bash
pip install ./dist/python
```

```python
from event_schema import Event

event = Event.model_validate(incoming)  # raises on contract violation
```

## Regenerating the Python models

Requires Node 20+ and Python 3.11+.

```bash
make python-codegen
```

This runs the TS build, then `datamodel-codegen` over the JSON Schemas, then the RootModel fixup.
`make schema-codegen-clean` removes the codegen venv and generated Python.

## Layout

```
src/
├── primitives.ts    shared scalar/enum building blocks
├── payloads.ts       per-event-type payload schemas
├── event.ts          the Event discriminated union
├── checkpoint.ts     DOM checkpoint schema
├── version.ts        schema version constant
└── index.ts          public exports
scripts/
├── build.ts               Zod -> JSON Schema
└── fix_rootmodel_config.py post-process for generated Pydantic
dist/                  generated JSON Schema + Pydantic package (committed)
test/                  47 tests, 100% coverage
```
